# Clearing a secret-scan finding safely

Covers what to do when the cs-webinars gitleaks scan flags a value, and the separation of duties that keeps the PR author from waving their own finding through. Mirrors the repo's `docs/overriding-a-safe-secret-scan.md` — if the two disagree, the repo wins.

## Treat every finding as real first

The scan is a backstop. A finding is a real leaked credential until you have proven it is not. Do not reach for the allowlist as the default response to a red scan — most red scans mean "remove this," not "allowlist this."

## The three-part safety test

A flagged value may be allowlisted **only if all three are true**:

1. It grants access to nothing real — no production system, no account, no paid service.
2. It is sample material that attendees regenerate or replace when they run the lab.
3. It would cost nothing if a stranger copied it straight out of the repo.

A throwaway self-signed cert the lab tells people to regenerate passes all three. A "test" token that still works against a real API does not — remove it and rotate it. If you cannot honestly say yes to all three, it is not safe: remove the value, and rewrite history if it already landed in a commit.

## The allowlist entry (the sanctioned mechanism)

Once a value passes the test, add a **scoped** `[[allowlists]]` block to `.gitleaks.toml` at the repo root, **in the same PR** that introduces the file. Match a single file path — not a directory, not a pattern that could swallow future secrets — and write a description saying *why* the value is safe, so the next reader doesn't have to re-derive it.

Template (the Hardening Konnect lab cert is the worked example already in the file):

```toml
[[allowlists]]
description = """
Throwaway self-signed TLS material generated for the <lab name> lab.
The private key is regenerated by attendees when they run the lab; it grants
access to nothing and is published deliberately as sample material.
"""
paths = [
  '''^webinars/<slug>/path/to/file\.(key|cert)$''',
]
```

Add a **new** block alongside any existing ones. Do not widen an existing block's `paths` list — each exception stays tied to its own reason.

Verify locally before pushing:

```bash
gitleaks detect --source . --redact --no-banner
```

`no leaks found` and exit code `0` means the allowlist covers your file and the scan is otherwise clean.

## Separation of duties — author proposes, reviewer authorizes

This is the rule the user specifically asked this skill to hold to. When you are running this skill you are the **PR author**, and the author's authority stops at proposing:

- **Propose (author, allowed):** decide a value looks safe, draft the `.gitleaks.toml` allowlist entry with its justification, include it in the PR, and call it out explicitly in the PR description so a reviewer knows a scan exception is in play and can scrutinize it.
- **Authorize (reviewer only):** ratify that the value is genuinely safe and that the allowlist is scoped correctly. That judgment belongs to a reviewer, not the author. The reviewer's approval of the PR *is* the authorization.

Concretely, when running this skill, do not:

- merge a PR that adds an allowlist entry on your own say-so, and
- use a repo admin's "merge past a failed check" button to get around a red scan — even if the account has the rights.

## Why not just merge past it

A repo admin can merge past a failed check, and for a one-off it's tempting. Don't. The override lives only in that one merge: the finding comes back red on the next push to `main`, the next contributor hits a failure they can't explain, and nobody can tell an approved exception from an accident. A reviewed `.gitleaks.toml` entry is the opposite — it carries its own reason and keeps every later run green.

## Making it enforceable, not just conventional

The steps above are author-side discipline. If the user wants the author to be genuinely *unable* to bypass the scan (rather than just trusted not to), that's a repo-settings job the skill can't do on its own. Point them at these, on the `KongHQ-CX/cs-webinars` repo:

- **Branch protection on `main`:** require a pull request, require the secret-scan status check to pass, and require at least one approving review before merge.
- **Turn off admin bypass:** enable "Do not allow bypassing the above settings" (a.k.a. include administrators) so an admin can't merge past a red check.
- **CODEOWNERS on `.gitleaks.toml`:** make a maintainer a required reviewer for any change to `.gitleaks.toml`, so adding or widening an allowlist always pulls in someone other than the author.
- **Require review from someone other than the author** (GitHub enforces that an approval can't come from the PR author), which is what turns "propose vs. authorize" into a hard rule instead of a habit.

Setting these is a maintainer action outside this skill's scope — surface it as the way to make the rule real, and let the user decide.
